# CHANGELOG.md

Chronological ledger. Record why changes happened, not just what changed.

## 2026-05-01

- Added root project governance files: `DESIGN.md`, `AGENTS.md`, `TASKS.md`, and `CHANGELOG.md`. This turns the frozen prototype skeleton into an executable contract: shared C++ core first, verified TypeScript semantics before bindings, and Godot/PufferLib as consumers instead of parallel rule engines.
- Established the double-buffered cascade rule as a design invariant. Chain Reaction explosions must resolve in simultaneous waves because in-place scan-order mutation can bias outcomes based on iteration direction rather than board state.
- Locked in flat fixed-size board state as a performance constraint. PufferLib needs thousands of duplicated environments per CPU core; allocator-heavy or pointer-heavy layouts would spend the budget on memory mechanics instead of environment stepping.
- Preserved the staged pipeline order: TypeScript prototype, C++ header verification, Cython/PufferLib binding, training, then Godot binding. The current repository already contains early consumer stubs, but binding work must wait until the core header is mathematically verified.
- Folded the GDD into the design contract: 8x8 deterministic Atoms mechanics, PPO self-play with a history pool, illegal-action masking, tiny 2D CNN policy/value network, and temperature-scaled Godot inference. This replaced the earlier three-channel observation placeholder with the sharper signed-distance-to-explosion channel so the model receives board physics directly.
- Implemented matching TypeScript and C++ core rules for legal moves, critical mass, deterministic double-buffered cascades, ownership transfer, and winner detection. The provided implementation sketch was adjusted to preserve the opening transient: the first placed token is not a win in a two-player elimination game.
- Added a TypeScript/C++ fixture equivalence harness. This gives the project a cheap regression gate before Cython or Godot bindings are allowed to consume the core.
- Recorded `uv` as the mandatory Python toolchain. Training and Cython work must not depend on ambient Python, pip, or global site packages because reproducibility matters before speed claims mean anything.
- Added the first Cython bridge after the TypeScript/C++ equivalence gate passed. The bridge exposes the verified C++ header to Python without duplicating gameplay rules, and it is built/tested through `uv` so training work does not depend on ambient interpreter state.
- Replaced the stale `training/game.pyx` consumer that referenced `core-logic/chain_reaction.hpp`. Keeping a broken path around after the shared core moved to `core/chain_reaction.hpp` would invite accidental parallel plumbing.
- Replaced scan-order ownership overwrite with simultaneous pressure cancellation. Same-owner pressure stacks; opposing two-player pressure cancels to a net value, so engine iteration order no longer decides contested cells.
- Replaced turn-count winner gating with transition-based elimination state. This was first modeled with seen and alive masks, then simplified to alive-only once the game model settled on all participants starting alive and eliminations occurring only after explosion transitions.
- Added core-owned legal-action and signed-distance observation exports. This keeps training consumers from reimplementing legality, critical mass, or perspective math in Python.
- Simplified player lifecycle state to an alive mask initialized with all participating players. Winner is now derived from that mask rather than stored, and eliminations are recomputed only after explosion transitions.
- Clarified that simulation state and training observation are separate schemas. The core stores owner plus token count because transient overfull cells are valid during cascade resolution; signed-distance observation is only a derived stable-state view.
- Accepted unbounded cascade resolution for the MVP. Any future wave cap must be an explicit bounded-mode/error-state design, not a quiet physics change.
- Added explicit corner, edge, and center critical-mass fixtures. The MVP keeps the 8x8 chessboard shape, and these tests pin the geometry before renderer or training consumers build assumptions on top of it.
- Added a fixed-size per-move wave log to the shared core and Cython bridge. Rendering can now animate actual simultaneous cascade waves without re-simulating rules, while simulation remains unbounded; only the inspection log can truncate.

## 2026-05-02

- Rejected core draw/max-turn handling. Chain Reaction gameplay remains unbounded; training may use a high episode cap as harness-level truncation only, so batch management cannot silently become physics.
- Marked the corner, edge, and center critical-mass fixture tasks complete now that the latest verification work includes them.
- Briefly prototyped, then removed, a legacy PufferLib-style Python wrapper. PufferLib v4 does not expose that API, so keeping it would have advertised the wrong training seam.
- Clarified that turns are alternating while cascade waves are simultaneous. Opposing-pressure cancellation fixtures remain as mathematical symmetry checks for the resolver, even when ordinary stable-turn play rarely reaches those arbitrary wave states.
- Added a random legal self-play smoke test against the Cython core. The test samples only from core-owned legal masks and expects episodes to reach a real winner before the smoke-test harness cap.
- Split the Cython bridge into logged and fast stepping paths. Rendering/debug consumers can keep `step()` wave logs, while training uses `step_fast()` so rollout throughput does not pay for cascade inspection buffers.
- Changed cascade terminal semantics to stop after any simultaneous wave that leaves exactly one player with tokens. A legal random self-play rollout exposed a period-26 owner-only cascade after P2 had already been eliminated; winner detection now terminates the turn at elimination instead of requiring post-victory physics to stabilize.
- Replaced the provisional Python `PufferEnv` wrapper with a PufferLib v4 Ocean-style native environment scaffold. PufferLib v4 does not expose the old `pufferlib.emulation`/`PufferEnv` API; the trainable path is `ocean/<env>/binding.c` built with `build.sh chain_reaction --cpu` on macOS or the CUDA backend on the workstation.
- Chose workstation-first training as the next milestone. The M1 laptop remains a verification/smoke environment; the next useful product loop is a minimal CUDA/PufferTank training run, then a CLI/TUI policy replay viewer, then Godot once the policy loop proves life.

## 2026-05-03

- Added PufferTank 4.0 container wiring for the Fedora CUDA workstation. The compose setup mounts this repository into the PufferLib container and symlinks the Ocean environment/config at runtime, preserving `core/chain_reaction.hpp` as the shared source instead of copying gameplay code into the image.
- Aligned the workstation path with the official PufferTank runtime contract instead of treating the Puffertank repository as a local dependency. This repo's Compose wrapper mirrors `docker.sh` behavior around the `pufferai/puffertank:4.0` image, GPU access, host IPC/cgroup/networking, and optional display mounts, then adds only Chain Reaction's env/core symlinks.
- Added a bounded 10-minute training entrypoint. The run uses PufferLib v4 inside the container, builds the CUDA Ocean backend by default, writes checkpoints/logs under `training/`, and treats the timeout as a product probe rather than evidence of final policy strength.
- Fixed the Ocean environment's terminal-step contract. Terminal and illegal-move branches now reset board state for the next episode without clearing the reward/terminal signal that Puffer consumes for the just-finished step; the previous implementation logged `-1` internally but returned a cleared rollout signal.
- Verified the CUDA/PufferTank path with build-only, finite, and timeout-bounded training runs. A tiny finite run now exits cleanly and writes checkpoints/logs; a longer smoke shows nonzero policy/value losses and entropy movement, but action masking remains the next correctness cut because unmasked random play starts dominated by illegal moves.
- Chose repo-owned Torch PPO as the learning path instead of optimizing around PufferLib's native CUDA model/trainer. PufferLib remains the vectorized Ocean environment engine; this repository owns the model, logits masking, PPO buffer semantics, checkpoints, and inspection code so the RL loop stays understandable and portable to future games.
- Replaced the provisional flat/MLP model sketch with a tiny spatial residual CNN: one signed-distance input plane, 32-channel stem, four pre-activation GroupNorm/SiLU residual blocks, a per-cell `1x1` policy head, and a small global-average-pooled value head. The policy should preserve board/action alignment rather than flattening the board into a generic MLP before action scoring.
- Made current-player value semantics explicit. Observations are sign-flipped so the model never needs to know whether it is player A or B, but PPO bootstrapping must negate the next nonterminal value because the next state is from the opponent's perspective. This is a negamax value target, not ordinary single-agent GAE.
- Reaffirmed the signed-distance observation contract: `0` is empty, positive values are current-player cells measured as `critical_mass - tokens`, and negative values are opponent cells with the same distance. If the implementation emits token counts instead, that is a semantic regression to fix before model training.
- Corrected the TypeScript and C++ observation export to match the signed-distance contract. The previous implementation emitted `tokens - critical_mass`, which inverted opportunity/danger signs relative to the model design.
- Added the first repo-owned Torch PPO scaffold: a 32-channel residual CNN, legal-logit masking from observation sign, negamax GAE for alternating current-player turns, a PufferLib Ocean vector adapter, and a separate PufferTank entrypoint. A tiny one-update container run completed with zero illegal selected actions and finite PPO losses.
- Documented the Fedora local Cython bridge smoke path. The workstation needs `gcc-c++` for the `c++` driver, while Cython and Python headers come from `uv`; the bridge build must currently run from `training/` because `setup.py` declares `chain_reaction.pyx` as a cwd-relative source.
- Hardened the negamax GAE smoke fixtures with hand trajectories for terminal returns, nonterminal bootstrap sign alternation, and batched terminal reset masking. This pins the current-player value convention before longer training runs can hide a sign bug behind plausible losses.
- Ran a larger Torch PPO smoke through PufferTank: 1,048,576 agent steps over 64 updates with 512 agents and 32-step horizons. The run completed with zero illegal selected actions, finite PPO losses/KL, JSON logs, and per-update checkpoints under `training/checkpoints/torch_ppo/`.
- Mirrored the repo-owned Torch PPO model architecture into a Python reference (`training/torch_ppo/model.py`, `masking.py`, `gae.py`) and into native CUDA semantics via a hardcoded PufferLib fork. The native trainer runs the same legal mask (`obs >= 0`) and negamax GAE (`r - gamma*V(next)`, `-gamma*lambda*next_adv`) in `pufferlib.cu` kernels, verified at ~600K SPS with zero illegal moves measured on the env side.
- Forked PufferLib v4 as a repo submodule (`vendor/PufferLib`, branch `chain-reaction-native`) instead of runtime-patching the container's installed source. The submodule carries the two CUDA semantic patches hardcoded — no boolean flags, no config knobs — and the Compose wiring points PUFFER_ROOT at the submodule so the container builds from the fork.
- Removed `training/puffer_ocean/patch_pufferlib.py`, the runtime container source patch hack. This was a correct diagnosis but wrong boundary; the fork makes the dependency explicit and eliminates silent runtime divergence between what the host edits and what the container runs.
- Fixed Ocean env logging: added `--log.n` (episode count) and `--log.illegal_moves` to the PufferLib dict so the dashboard can compute per-episode averages and validate the legal action mask at env level. Illegal moves verified at 0.000.
- Changed the self-play reward convention to log both winner (+1) and loser (-1) as separate PufferLib entries per game. Previously only the winner's terminal step received an explicit reward; the loser's final nonterminal move had reward 0 and relied entirely on value bootstrap. The double-entry zero-sum scheme gives both sides explicit reward signals and makes episode_return = 0 / perf = 0.5 for balanced random play, with `winrate` (player 1 wins / games) as the actual skill-tracking stat.
- Published the PufferLib fork as a public repository: `github.com/Nelathan/chain-reaction-pufferlib`. The `.gitmodules` URL and submodule `origin` remote now point at the fork instead of upstream `PufferAI/PufferLib`; `upstream` remote tracks the original for future sync.

## 2026-05-06

- Reviewed the native PufferLib CUDA patch in the submodule instead of treating the successful training run as proof of correctness. The legal mask placement is coherent: rollout sampling, PPO logprob/entropy, and policy-gradient logits all ignore illegal opponent-owned cells via `obs >= 0`.
- Found a real negamax GAE sharp edge: PufferLib chooses scalar or vector advantage kernels based on horizon alignment, and the patched formulas were not algebraically identical once importance ratios differ from `1.0`. Fixed both paths to use the same Chain Reaction TD error, `rho * (r - gamma * V(next) - V(current))`, with the negamax recursive sign preserved.
- Removed noisy CUDA narrowing warnings in the native PPO observation-stride fields by making the intended cast explicit. This is not a gameplay semantic change; it makes PufferTank rebuild output easier to read.
- Re-verified through the PufferTank/Podman image, not the host toolchain. `BUILD_ONLY=1 podman compose -f compose.yaml -f compose.podman.yaml run --rm puffer` rebuilt `pufferlib/_C.cpython-312-x86_64-linux-gnu.so`; the tiny finite `8192` timestep smoke also completed with finite losses.
- Added root README and refreshed `AGENTS.md`, `DESIGN.md`, and `TASKS.md` so future sessions use the image-first PufferTank protocol, recognize generated submodule symlinks, and track the remaining need for a direct native advantage-kernel fixture.
- Clarified the container boundary: Compose plus bind mounts is the correct active development workflow because it rebuilds against the current checked-out submodule files. A derived Dockerfile from `pufferai/puffertank:4.0` is still useful later for immutable CI/release packaging, but should wait until the native CUDA semantics are fixture-verified.
- Audited the actual native model path and found that PufferLib native is not training the repo-owned Torch residual CNN. It uses the default native policy: linear encoder over 64 flat features, one MinGRU layer, linear decoder, 64-action head, scalar value, 65,664 parameters, and no convolution kernels. Runtime logging now prints this so stale docs cannot impersonate reality.
- Benchmarked short native jobs through PufferTank. A 262,144-step Chain Reaction run with 4096 agents, horizon 64, minibatch 32768, and max_turns 64 reported about 4.1M SPS, with rollout/env time dominating after the first epoch; native cartpole under a comparable batch shape reported about 8.3M SPS. This points at Chain Reaction environment/cascade work as a real cost, not at a dead GPU or missing CUDA path.
- Restored minimum durable native telemetry: JSON logs now include checkpoint path; the Ocean env logs truncations plus average cascade depth from a lean core-owned stats path that does not allocate and does not copy the full wave flight recorder during training.
- Verified that a saved native checkpoint can be loaded by `pufferlib.pufferl eval`; the command loaded weights successfully but the stock eval path enters an open-ended render/rollout loop, so policy strength remains unevaluated. A non-rendering evaluator against random legal play is now the next binary gate before calling any checkpoint playable.
- Confirmed current native self-play is single-policy only. There is no history pool, no sampled checkpoint opponents, and no frozen opponent weights during rollout. History-pool work should wait until a loaded checkpoint beats random legal play by a measurable margin.
- Added a finite, non-rendering native checkpoint evaluator against random legal play. It rebuilds the Chain Reaction Ocean env inside PufferTank, loads native `.bin` weights, masks checkpoint logits with `obs >= 0`, samples the random opponent from the same legal mask, and writes JSON reports under `training/evals/chain_reaction/`.
- Killed a stale `pufferlib.pufferl eval chain_reaction --render-mode None` container that was burning CPU in the stock infinite eval loop. This path is now documented as a trap: it proves checkpoint loading but is not a finite evaluator.
- Evaluated `training/checkpoints/chain_reaction/1778090124759/0000000000262144.bin` for 1000 deterministic games against random legal play at `max_turns=512`: combined winrate `0.584`, P1 winrate `0.600`, P2 winrate `0.568`, illegal moves `0`, truncations `0`, mean episode length `120.344`, median `120.0`, mean cascade depth `1.0245`, mean terminal reward `0.168`. This is better than random but not yet strong enough to call confidently playable; inspect reward/value/entropy targets before longer or more elaborate native training.
- Interpreted the 58.4% checkpoint result as evidence of learning, not readiness. The next cut is checkpoint-progression evaluation over a longer native single-policy run; if winrate climbs with steps, the default native model is viable for a first dumb AI, while a plateau around 55–60% should trigger architecture/history-pool work instead of premature complexity. Also promoted the direct `forward_policy` / `_C.close(policy)` segfault from an acceptable short-lived evaluator workaround to an explicit tracked task.
- Corrected course after finding the train/eval cap mismatch: the evaluated checkpoint was trained at `max_turns=64` but evaluated at `512`, and the saved native metrics showed the 64-turn runs ending entirely in truncation/tie rewards. Reworked Ocean logging so `env/n` counts real game endings instead of fake zero-sum log entries, episode length is counted once per game, and terminal/truncation rates are exposed directly. PufferTank telemetry smokes now make the boundary visible: `max_turns=64` produced terminal_rate `0.0` / truncation_rate `1.0`, while `max_turns=128` produced terminal_rate `0.992` / truncation_rate `0.008` with mean episode length `120.793`. The next training cap should be at least `128`; treating the 64-turn checkpoint as strength evidence was invalid.
- Re-evaluated the same 64-turn-trained checkpoint under matched and near-matched caps. At `max_turns=64`, it produced combined winrate `0.0` with `1000/1000` truncations, exactly matching the poisoned training cap. At `max_turns=128`, it produced combined winrate `0.575` with `12/1000` truncations and mean episode length `120.325`. This confirms the earlier `512`-cap strength read was mostly measuring behavior outside the training terminal horizon; future native runs must train with a cap that actually reaches terminal games before checkpoint strength comparisons mean anything.
- Ran a valid `max_turns=128`, horizon `128`, 1,048,576-step native smoke after rejecting a too-small 262,144-step attempt that was less than one full rollout epoch for 4096 agents. The valid run finished with finite losses, terminal_rate `0.984`, truncation_rate `0.016`, illegal_moves `0`, mean episode length `120.737`, and checkpoint `training/checkpoints/chain_reaction/1778096039119/0000000001572864.bin`. Matched-cap evaluation over 1000 games scored combined winrate `0.579` (`P1=0.564`, `P2=0.594`) with `15` truncations. This shows the corrected cap produces real terminal training data, but the default native model still looks like a weak local-competence baseline, not the intended CNN.

## 2026-05-07

- Added a reproducible native PufferLib CNN encoder checkpoint in the submodule and advanced the root submodule pointer so clean checkouts no longer depend on an untracked `chain_reaction_cnn.cu` ghost file. This fixed repository integrity, but it also exposed the wrong iteration surface: every Chain Reaction model change inside PufferLib requires submodule commits, root pointer updates, and native CUDA rebuild/debug loops.
- Defaulted the PufferTank native build to PufferLib `--float`/fp32 for debugging. The reason is tactical: PPO, value bootstrapping, legal masks, custom CUDA kernels, and architecture experiments should not be debugged through bf16 noise. This is not a claim that fp32 is the final throughput choice; bf16 should be an ablation after the learning loop is stable.
- Paused native PufferLib model work as the primary architecture path. The native CNN experiment is smaller than the intended design model and still feeds PufferLib's MinGRU/default decoder instead of spatial policy/value heads. Continuing to force the exact design CNN through PufferLib internals is too slow and too opaque for the current milestone.
- Re-centered training work on `training/torch_ppo/`, where the exact documented 32-channel, four-block GroupNorm/SiLU residual CNN already lives in ordinary PyTorch. PufferLib remains useful as the Ocean environment engine and as a later acceleration target, but Chain Reaction model iteration should happen in the root repo until the Torch model learns measurably.
