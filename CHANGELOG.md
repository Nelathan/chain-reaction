# CHANGELOG.md

Chronological ledger. Record why changes happened, not just what changed.

## 2026-05-01

- Added root project governance files: `DESIGN.md`, `AGENTS.md`, `TASKS.md`, and `CHANGELOG.md`. This turns the frozen prototype skeleton into an executable contract: shared C++ core first, verified TypeScript semantics before bindings, and Godot/PufferLib as consumers instead of parallel rule engines.
- Established the double-buffered cascade rule as a design invariant. Chain Reaction explosions must resolve in simultaneous waves because in-place scan-order mutation can bias outcomes based on iteration direction rather than board state.
- Locked in flat fixed-size board state as a performance constraint. PufferLib needs thousands of duplicated environments per CPU core; allocator-heavy or pointer-heavy layouts would spend the budget on memory mechanics instead of environment stepping.
- Preserved the staged pipeline order: TypeScript prototype, C++ header verification, Cython/PufferLib binding, training, then Godot binding. The current repository already contains early consumer stubs, but binding work must wait until the core header is mathematically verified.
- Folded the GDD into the design contract: 8x8 deterministic Atoms mechanics, PPO self-play with a history pool, illegal-action masking, tiny 2D CNN policy/value network, and temperature-scaled Godot inference. This replaced the earlier three-channel observation placeholder with the sharper signed-distance-to-explosion channel so the model receives board physics directly.
- Implemented matching TypeScript and C++ core rules for legal moves, critical mass, deterministic double-buffered cascades, ownership transfer, and winner detection. The provided implementation sketch was adjusted to preserve the opening transient: the first placed token is not a win in a two-player elimination game.
- Added a TypeScript/C++ fixture equivalence harness. This gives the project a cheap regression gate before Cython or Godot bindings are allowed to consume the core.
- Recorded `uv` as the mandatory Python toolchain. Training and Cython work must not depend on ambient Python, pip, or global site packages because reproducibility matters before speed claims mean anything.
- Added the first Cython bridge after the TypeScript/C++ equivalence gate passed. The bridge exposes the verified C++ header to Python without duplicating gameplay rules, and it is built/tested through `uv` so training work does not depend on ambient interpreter state.
- Replaced the stale `training/game.pyx` consumer that referenced `core-logic/chain_reaction.hpp`. Keeping a broken path around after the shared core moved to `core/chain_reaction.hpp` would invite accidental parallel plumbing.
- Replaced scan-order ownership overwrite with simultaneous pressure cancellation. Same-owner pressure stacks; opposing two-player pressure cancels to a net value, so engine iteration order no longer decides contested cells.
- Replaced turn-count winner gating with transition-based elimination state. This was first modeled with seen and alive masks, then simplified to alive-only once the game model settled on all participants starting alive and eliminations occurring only after explosion transitions.
- Added core-owned legal-action and signed-distance observation exports. This keeps training consumers from reimplementing legality, critical mass, or perspective math in Python.
- Simplified player lifecycle state to an alive mask initialized with all participating players. Winner is now derived from that mask rather than stored, and eliminations are recomputed only after explosion transitions.
- Clarified that simulation state and training observation are separate schemas. The core stores owner plus token count because transient overfull cells are valid during cascade resolution; signed-distance observation is only a derived stable-state view.
- Accepted unbounded cascade resolution for the MVP. Any future wave cap must be an explicit bounded-mode/error-state design, not a quiet physics change.
- Added explicit corner, edge, and center critical-mass fixtures. The MVP keeps the 8x8 chessboard shape, and these tests pin the geometry before renderer or training consumers build assumptions on top of it.
- Added a fixed-size per-move wave log to the shared core and Cython bridge. Rendering can now animate actual simultaneous cascade waves without re-simulating rules, while simulation remains unbounded; only the inspection log can truncate.

## 2026-05-02

- Rejected core draw/max-turn handling. Chain Reaction gameplay remains unbounded; training may use a high episode cap as harness-level truncation only, so batch management cannot silently become physics.
- Marked the corner, edge, and center critical-mass fixture tasks complete now that the latest verification work includes them.
- Briefly prototyped, then removed, a legacy PufferLib-style Python wrapper. PufferLib v4 does not expose that API, so keeping it would have advertised the wrong training seam.
- Clarified that turns are alternating while cascade waves are simultaneous. Opposing-pressure cancellation fixtures remain as mathematical symmetry checks for the resolver, even when ordinary stable-turn play rarely reaches those arbitrary wave states.
- Added a random legal self-play smoke test against the Cython core. The test samples only from core-owned legal masks and expects episodes to reach a real winner before the smoke-test harness cap.
- Split the Cython bridge into logged and fast stepping paths. Rendering/debug consumers can keep `step()` wave logs, while training uses `step_fast()` so rollout throughput does not pay for cascade inspection buffers.
- Changed cascade terminal semantics to stop after any simultaneous wave that leaves exactly one player with tokens. A legal random self-play rollout exposed a period-26 owner-only cascade after P2 had already been eliminated; winner detection now terminates the turn at elimination instead of requiring post-victory physics to stabilize.
- Replaced the provisional Python `PufferEnv` wrapper with a PufferLib v4 Ocean-style native environment scaffold. PufferLib v4 does not expose the old `pufferlib.emulation`/`PufferEnv` API; the trainable path is `ocean/<env>/binding.c` built with `build.sh chain_reaction --cpu` on macOS or the CUDA backend on the workstation.
- Chose workstation-first training as the next milestone. The M1 laptop remains a verification/smoke environment; the next useful product loop is a minimal CUDA/PufferTank training run, then a CLI/TUI policy replay viewer, then Godot once the policy loop proves life.

Terminal and illegal-move branches now reset board state for the next episode without clearing the reward/terminal signal that Puffer consumes for the just-finished step; the previous implementation logged `-1` internally but returned a cleared rollout signal.

A tiny finite run now exits cleanly and writes checkpoints/logs; a longer smoke shows nonzero policy/value losses and entropy movement, but action masking remains the next correctness cut because unmasked random play starts dominated by illegal moves.

- Replaced the provisional flat/MLP model sketch with a tiny spatial residual CNN: one signed-distance input plane, 32-channel stem, four pre-activation GroupNorm/SiLU residual blocks, a per-cell `1x1` policy head, and a small global-average-pooled value head. The policy should preserve board/action alignment rather than flattening the board into a generic MLP before action scoring.
- Made current-player value semantics explicit. Observations are sign-flipped so the model never needs to know whether it is player A or B, but PPO bootstrapping must negate the next nonterminal value because the next state is from the opponent's perspective. This is a negamax value target, not ordinary single-agent GAE.
- Reaffirmed the signed-distance observation contract: `0` is empty, positive values are current-player cells measured as `critical_mass - tokens`, and negative values are opponent cells with the same distance. If the implementation emits token counts instead, that is a semantic regression to fix before model training.
- Corrected the TypeScript and C++ observation export to match the signed-distance contract. The previous implementation emitted `tokens - critical_mass`, which inverted opportunity/danger signs relative to the model design.
- Added the first repo-owned Torch PPO scaffold: a 32-channel residual CNN, legal-logit masking from observation sign, negamax GAE for alternating current-player turns, a PufferLib Ocean vector adapter, and a separate PufferTank entrypoint. A tiny one-update container run completed with zero illegal selected actions and finite PPO losses.
- Documented the Fedora local Cython bridge smoke path. The workstation needs `gcc-c++` for the `c++` driver, while Cython and Python headers come from `uv`; the bridge build must currently run from `training/` because `setup.py` declares `chain_reaction.pyx` as a cwd-relative source.
- Hardened the negamax GAE smoke fixtures with hand trajectories for terminal returns, nonterminal bootstrap sign alternation, and batched terminal reset masking. This pins the current-player value convention before longer training runs can hide a sign bug behind plausible losses.
- Ran a larger Torch PPO smoke through PufferTank: 1,048,576 agent steps over 64 updates with 512 agents and 32-step horizons. The run completed with zero illegal selected actions, finite PPO losses/KL, JSON logs, and per-update checkpoints under `training/checkpoints/torch_ppo/`.

The native trainer runs the same legal mask (`obs >= 0`) and negamax GAE (`r - gamma*V(next)`, `-gamma*lambda*next_adv`) in `pufferlib.cu` kernels, verified at ~600K SPS with zero illegal moves measured on the env side.
- Forked PufferLib v4 as a repo submodule (`vendor/PufferLib`, branch `chain-reaction-native`) instead of runtime-patching the container's installed source. The submodule carries the two CUDA semantic patches hardcoded — no boolean flags, no config knobs — and the Compose wiring points PUFFER_ROOT at the submodule so the container builds from the fork.
- Removed `training/puffer_ocean/patch_pufferlib.py`, the runtime container source patch hack. This was a correct diagnosis but wrong boundary; the fork makes the dependency explicit and eliminates silent runtime divergence between what the host edits and what the container runs.
- Fixed Ocean env logging: added `--log.n` (episode count) and `--log.illegal_moves` to the PufferLib dict so the dashboard can compute per-episode averages and validate the legal action mask at env level. Illegal moves verified at 0.000.
- Changed the self-play reward convention to log both winner (+1) and loser (-1) as separate PufferLib entries per game. Previously only the winner's terminal step received an explicit reward; the loser's final nonterminal move had reward 0 and relied entirely on value bootstrap. The double-entry zero-sum scheme gives both sides explicit reward signals and makes episode_return = 0 / perf = 0.5 for balanced random play, with `winrate` (player 1 wins / games) as the actual skill-tracking stat.
- Published the PufferLib fork as a public repository: `github.com/Nelathan/chain-reaction-pufferlib`. The `.gitmodules` URL and submodule `origin` remote now point at the fork instead of upstream `PufferAI/PufferLib`; `upstream` remote tracks the original for future sync.

- PufferVec is destroyed and recreated on each size switch to set the new `active_width`/`active_height`. Cost is ~10ms per switch vs ~2800ms PPO update — negligible overhead.
- Default rollout geometry tightened: horizon 32 (was 128), 1024 agents, ~32K impressions per update, ~1000 updates in 32M total impressions.
- Eval runs every 100 updates on the current training size (32 games). Full sweep eval across all unlocked sizes every 500 updates (64 games each). Final sweep across all sizes at training end.
- Simplified `curriculum.sh` to single Ocean compilation + single `train.py` invocation. Removed `PROMOTION_GATE`, per-size shell loops, and checkpoint-detection logic.
- Removed `promotion_gate` from `TrainConfig`. Replaced `active_width`/`active_height` config fields with scheduler-driven dynamic switching.

- Re-ran the native `6x6` Torch PPO path after removing the curriculum plumbing: the same 10M-step settings (`total_agents=1024`, `horizon=128`, `minibatch_size=8192`, `max_turns=128`) still reached `combined_winrate=1.0` at eval update `77` with zero illegal selected actions and zero truncations. A fresh native `4x4` run under matched settings (`max_turns=32`) also reached `combined_winrate=1.0` at eval update `77` with zero illegal selected actions and zero truncations. These are strength-and-plumbing sanity checks, not proof of exact repeatability.

## 2026-05-09

- Reduced PPO hot-path overhead by removing `Categorical` construction from the training loop and reusing a precomputed legal-action mask across rollout and update. That trimmed some tensor churn, but the real throughput lever was minibatch granularity: `minibatch_size=32768` on the 1024-agent, 128-horizon 8x8 run shape raised steady-state throughput to about `43.4K SPS` with `update_ms` around `1.8s`, while `65536` OOMed on this GPU. The long-run default now reflects the safe faster setting.
- Tightened the Torch PPO profiling contract after finding that several `CHAIN_REACTION_*` overrides were not being passed through the Compose launcher. The wrapper now forwards log/eval/compile/sync/W&B controls, and `CHAIN_REACTION_SYNC_TIMING=1` can opt into CUDA-synchronized phase timing without adding default hot-loop synchronizations.
- Changed the Torch PPO default to one PPO update epoch. Synchronized timing showed the update phase dominated the loop (`~2.8s` update vs `~0.3s` rollout on 8x8/1024/128), and dropping from two epochs to one cut update time to about `1.4s` and short-run throughput to about `75K SPS`. A 10M-step 4x4 sanity run reached `combined_winrate=1.0` by update 25; a matched 6x6 run reached `0.90625` by update 25 and `1.0` by update 50, with zero illegal selected actions and zero truncations. This is a faster baseline, not proof that one epoch is optimal for final 8x8 policy quality.
- Made the Torch CNN input geometry-aware and canonical. The model now expands the signed-distance observation into four planes: normalized capacity, normalized own-token count, normalized opponent-token count, and signed closeness to critical. Old one-plane checkpoint compatibility was intentionally removed, and local checkpoint directories were cleared so new runs start from the current architecture instead of carrying stale model-shape baggage.
- Simplified the Torch CNN trunk from four GroupNorm pre-activation residual blocks to three plain residual blocks (`Conv3x3 -> SiLU -> Conv3x3 -> add`). The intent is to keep the model spatial and residual without importing normalization/depth ceremony before evidence says PPO needs it.
- Validated the simplified model on 6x6 with default settings: 10M steps, `SPS=~242K` (60% faster than the old GroupNorm trunk at `~151K`), `update_ms=15`, `combined_winrate=1.0` at update 77, zero illegal actions, zero truncations. The cleaner model learned faster and more aggressively: median episode length fell from 55 to 3.5 moves.
- Swapped the optimizer from `torch.optim.AdamW` to FlashOptim's `FlashAdamW` with native bf16 training/evaluation models. FLOPs are identical, but FlashOptim's fused Triton update kernel replaces the PyTorch CUDA AdamW kernel. Optimizer states are quantized to 8-bit with 24-bit master weight semantics via error correction terms. Checkpoints serialize as FP32 via `get_fp32_model_state_dict` and restore via `set_fp32_model_state_dict`. No `torch.amp` or `GradScaler` overhead.
